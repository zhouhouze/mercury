# 设置需求 PRD（v0）

## 1. 背景
Navia 需要一个足够轻量、足够稳定的设置能力，帮助用户快速完成 AI 能力接入并进入主功能使用。  
在 v0 阶段，设置页的目标不是做成通用配置中心，而是让用户以最少步骤完成 DeepSeek 的基础配置。

## 2. 目标
- 让用户快速完成可用的 LLM 配置。
- 降低首次配置门槛，避免 provider 和 model 选择带来的复杂度。
- 保证 Chatbot / Chat / Mindmap 等 LLM 功能能够基于同一套配置稳定工作。
- 让配置具备可保存、可恢复、可校验的基础能力。

## 3. 范围
### 3.1 v0 范围内
- 仅支持 DeepSeek。
- 仅支持一个默认 model。
- 支持 API Key 配置。
- 支持 model 输入与保存。
- 支持配置恢复。
- 支持基础连通性测试。
- 支持设置页展示当前配置状态。

### 3.2 v0 范围外
- 不支持 provider 切换。
- 不支持多 model 选择。
- 不支持模型推荐列表。
- 不支持会话级 provider / model 覆盖。
- 不支持多供应商配置管理。
- 不支持高级偏好中心化管理。

## 4. 用户故事
### 4.1 首次配置
用户第一次打开设置页时，希望只看到最少的配置项，并能快速填完 DeepSeek 的接入信息。

### 4.2 重新进入
用户刷新或重新打开设置页后，希望之前填写的配置能够恢复。

### 4.3 校验配置
用户填写完配置后，希望能够测试当前配置是否可用，避免进入主功能后才发现无法连接。

## 5. 功能需求
### 5.1 Provider
- 设置页仅展示 DeepSeek。
- 不展示 provider 切换入口。
- 不允许用户在设置页选择其他 provider。

### 5.2 Model
- 设置页提供单一 `model` 输入框。
- `model` 默认值为 `deepseek-v4-pro`。
- 用户可以手动修改该值。
- 不展示 model 下拉列表。
- 不展示多 model 推荐项。

### 5.3 API Key
- 设置页提供 `API Key` 输入框。
- `API Key` 为必填项。
- API Key 保存后可恢复到配置中。
- 保存后在界面中不应明文完整回显。

### 5.4 保存与恢复
- 用户修改配置后可保存。
- 页面刷新后应能恢复上次保存的配置。
- 恢复逻辑应保持当前设置页可继续编辑。

### 5.5 Test Connection
- 用户可对当前 DeepSeek 配置执行连通性测试。
- 测试结果应清楚反馈成功或失败。
- 测试失败时不清空用户已填写内容。

### 5.6 与主功能联动
- Chatbot、Chat、Mindmap 等功能默认使用同一套设置页保存的配置。
- v0 不支持这些功能内部单独切换 provider 或 model。

## 6. 业务规则
- v0 仅支持 DeepSeek。
- `model` 默认值为 `deepseek-v4-pro`。
- `API Key` 是最小必填项。
- `model` 允许手动编辑，但不提供候选项。
- 任何未配置完整的情况都不应被视为可用状态。
- 配置缺失时，LLM 请求应被阻止，并给出明确提示。
- 不允许在设置页出现 provider / model 管理复杂化入口。

## 7. 交互原则
- 首屏尽量简洁，只保留核心字段。
- 优先让用户完成配置，而不是理解配置体系。
- 错误提示要明确，避免让用户误以为是网络问题。
- 配置成功后状态应清晰可见。

## 8. 验收标准
- 用户首次进入设置页时，只看到 DeepSeek 的基础配置。
- 用户可填写 `API Key` 和 `model`。
- `model` 默认值为 `deepseek-v4-pro`。
- 用户可完成保存并在刷新后恢复。
- 用户可执行连通性测试并看到明确结果。
- 设置页不出现其他 provider 入口。
- 设置页不出现 model 下拉组件。
- LLM 功能在缺少配置时不会静默失败。

## 9. 风险与约束
- v0 功能较小，但这是刻意设计，用于降低首次使用门槛。
- 单模型策略会限制灵活性，但能提升稳定性和一致性。
- 后续如果扩展多 provider / 多 model，需要作为独立阶段演进，避免污染 v0 结构。

## 10. 后续演进方向
- v1: 引入 provider 管理与 model 推荐列表。
- v2: 支持 Chatbot 内会话级 provider / model 切换。
- v3: 演进为统一偏好中心，承载更多全局设置。
