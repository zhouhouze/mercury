# V1.3 Evidence Card Mindmap Stage Gate

状态：Conditional Go for V1.3-0 external audit
阶段定位：V1.2 AI Reading mock-first product path complete 之后的 Mindmap 体验升级阶段
日期：2026-06-16

---

## 1. 阶段目标

V1.3 的目标是把当前偏 Mermaid 默认渲染的思维导图体验，升级为 Navia 自有的 `Evidence Card Mindmap` 主渲染体验。

本阶段完成后，用户应能在 Chrome 原生 Side Panel 中看到一张高质量、可读性强、交互反馈明确、来源证据可追溯的思维导图：

```text
读取当前网页
-> A 产出 high-signal perception / digest / sourceRefs / qualityReport
-> C 产出 digest-first mindmap tree + nodeSourceMap
-> D 产出 mindmap Artifact / ToolResult / Trace
-> B 将 artifact 渲染为 Evidence Card Mindmap
-> 用户点击节点
-> 节点选中、邻接关系高亮、source evidence card 展开
-> 用户可触发 DOM jumpback 或查看 fallback evidence
```

V1.3 不声明完整 V1 complete，不声明 V2 memory / RAG ready，不新增 Web Research、PPT、Deep Research、Voice、桌宠、多 Agent 或浏览器自动操作能力。

## 2. 架构边界

### 2.1 模块职责

| 模块 | V1.3 职责 | 不允许做 |
|---|---|---|
| A Page Reading | 继续提供 `PerceptionDigest`、`SourceRef`、`QualityReport` | 不生成 Mindmap、不生成 Artifact、不发 SSE |
| C Mindmap | 继续生成 tree、Mermaid source、`nodeSourceMap`、node bindings | 不渲染 UI、不读取 DOM、不调用 content script |
| D Adapter Layer | 继续作为 ToolResult / Artifact / Event / Trace 唯一出口 | 不让 A/C/B 绕过治理写 Artifact/Event |
| B Mindmap Renderer | 新增 Evidence Card Mindmap 视图、交互状态、source card、fallback | 不直接调用 A/C/D 服务，不生成事实内容 |
| Content Script | 只执行用户触发的 source jumpback / highlight | 不做自动浏览器任务 |

### 2.2 渲染策略

V1.3 采用双轨渲染：

```text
主渲染：Evidence Card Mindmap
Fallback：Mermaid source / Mermaid visual / source text fallback
```

Evidence Card Mindmap 必须优先消费既有 `ArtifactRecord(type="mindmap")` 与 `metadata.nodeSourceMap`，不得要求 C 直接返回前端专用字段才能工作。若现有 artifact 缺少足够 metadata，B 必须显示明确降级原因。

### 2.3 合同冻结口径

`EvidenceCardViewModel` 是 B 模块本地渲染 view model，不是新的 Runtime public contract。它必须从 `ArtifactRecord(type="mindmap")`、`metadata.nodeSourceMap`、`metadata.nodeBindings`、`sourceRefIds`、`textQuote` 和 `fallbackText` 派生。

V1.3-0 必须冻结：

- `docs/active/project/contracts/v1_3_evidence_card_mindmap.schema.json`
- `EvidenceCardViewModel` 字段、状态和降级语义。
- V1.3 `report.json`、截图 metadata、HTML 报告证据结构。
- fixture class gate 和 No-Go 口径。

A/C/D 不得依赖 `EvidenceCardViewModel` 的 exact shape。若未来要让 C/D 直接消费该结构，必须回到合同冻结阶段。

## 3. 用户体验要求

### 3.1 Evidence Card 节点

每个主要节点应以卡片形式展示：

- 节点标题。
- 一句话摘要或 note。
- 来源数量。
- 质量 / 置信提示。
- 标签或节点类型。
- 可点击选中态。
- 可见 hover / focus / selected 状态。

### 3.2 交互反馈

必须支持：

- 点击节点后显示 source evidence panel。
- 选中节点高亮，非相关节点降低视觉权重。
- 相关边线和邻接节点高亮。
- source evidence panel 展示 `textQuote` 或 `fallbackText`。
- DOM jumpback 成功时展示定位成功状态。
- DOM jumpback 失败时展示 fallback evidence，不得伪装为成功。
- Mermaid 渲染失败时显示 source fallback。

### 3.3 视觉质量

目标质感：

- 安静、清晰、信息密度高。
- 卡片层级分明，不使用大面积单一色块。
- 连接线有方向感和结构感。
- 适配 Chrome Side Panel 窄宽度。
- 文本不溢出、不遮挡、不依赖负 letter-spacing。
- 动效短促、明确，不影响阅读。

## 5. 验收门槛

必须通过：

- [ ] Evidence Card Mindmap 是主视图，Mermaid 只是 fallback / source / debug。
- [ ] `EvidenceCardViewModel` 通过 schema validation，并且由 Artifact + metadata 派生，不要求 C 输出前端组件结构。
- [ ] B 不直接调用 A/C/D 服务，也不生成事实内容。
- [ ] 每个主要节点能追踪到 `nodeSourceMap` 或显示明确缺失原因。
- [ ] 每个可反跳节点展示 `sourceRefIds`、`textQuote` 或 `fallbackText`。
- [ ] 点击节点后有明显 selected state。
- [ ] 邻接节点或连接线有明确高亮 / 降权反馈。
- [ ] source evidence panel 在窄 Side Panel 中可读。
- [ ] DOM jumpback success 与 fallback shown 在 UI 和报告中严格区分。
- [ ] 长标题、重复标题、无 sourceRef、低信号页面都有 fixture 覆盖。
- [ ] 至少 8 个真实网页或可复现 snapshot 验收，其中至少 3 个真实 Chrome 原生 Side Panel 截图级样本。
- [ ] 生成 HTML 验收报告，展示目标架构、当前实现、用户路径截图、每页结果和 No-Go 结论。
- [ ] 生成 `docs/active/project/evidence/v1_3_evidence_card_mindmap/report.json`，字段符合 V1.3 schema。
