# V1.4 Reading Map Stage Gate

Status: Active documentation gate for staged implementation
Date: 2026-06-24

---

## 1. Stage Goal

V1.4 turns the V1.3 Evidence Card Mindmap into a Side Panel reading map. The user should be able to use the mindmap as navigation while reading:

```text
Current page -> PageContext -> Mindmap Artifact
-> EvidenceCardViewModel
-> Reading Map two-column view
-> select theme / node
-> see node detail, source evidence, and jumpback/fallback status
```

This stage does not introduce Canvas Knowledge Map, multi-page knowledge graph, Memory, RAG, Web Research, PPT, Deep Research, multi-agent orchestration, voice, desktop pet, browser automation product features, or default local file access.

Allowed claim after all gates pass:

```text
V1.4 Reading Map Side Panel navigation experience complete.
```

Not allowed:

```text
Full V1 complete.
Canvas Knowledge Map complete.
V2 Memory / RAG ready.
Web Research / PPT / Deep Research ready.
```

---

## 2. Architecture Boundary

| Module | V1.4 responsibility | No-Go |
|---|---|---|
| A Page Reading | Continue producing page perception, digest, source refs, and quality report | Do not generate Reading Map UI |
| C Mindmap | Continue producing mindmap tree, Mermaid source, node source map, and node bindings | Do not output React / DOM / CSS structures |
| D Adapter Layer | Continue owning ToolResult, Artifact, Event, and Trace output | Do not let A/C/B write governed records directly |
| B Renderer | Render a B-local Reading Map from EvidenceCardViewModel | Do not call A/C/D services directly and do not generate factual content |
| Content Script | User-triggered source jumpback / highlight only | Do not perform autonomous browser operations |

V1.4 must reuse the V1.3 B-local `EvidenceCardViewModel`. If additional UI-only fields are needed, they must remain B-local. Public runtime contracts remain unchanged.

---

## 3. User Experience Requirements

The Reading Map must be a compact two-column Side Panel experience:

- Left column: theme / node list derived from Evidence Card themes.
- Right column: selected node detail, source count, quality state, text quote or fallback evidence.
- Selecting a theme or node updates selected / neighbor / dimmed states.
- Source jumpback reuses V1.2 / V1.3 source path and clearly distinguishes located, fallback shown, and blocked.
- Narrow Side Panel width remains readable; no text overlap or hidden required controls.
- Mermaid remains debug / fallback, not the primary V1.4 navigation surface.

## 5. Acceptance Gate

Required:

- [ ] V1.4-0 documentation audit has no fatal or major issue.
- [ ] Reading Map is derived from existing `EvidenceCardViewModel`.
- [ ] B does not change Runtime public contracts and does not call A/C/D services directly.
- [ ] Left map column and right detail/evidence column are visible in Side Panel.
- [ ] Selecting a node updates selected state and detail pane.
- [ ] Missing source displays degraded / fallback reason instead of normal success.
- [ ] Source jumpback success and fallback shown are visibly and semantically distinct.
- [ ] Unit/component tests cover normal, missing source, duplicate label, long text, and dense theme cases.
- [ ] Real Chrome or reproducible snapshot acceptance runs against current Side Panel path.
- [ ] Acceptance evidence includes report, PRD review, false-green audit, and at least one screenshot-level artifact.

No-Go:

- [ ] Claiming V1.4 complete by only changing Mermaid CSS.
- [ ] Requiring C to output frontend component structures.
- [ ] Treating fallback evidence as DOM highlight success.
- [ ] Using fullscreen extension page evidence as native Side Panel proof without marking it as non-native.
- [ ] Expanding this stage into Canvas, Memory, RAG, Web Research, PPT, Deep Research, multi-agent, voice, desktop pet, browser automation, or default local file access.
