## 1. 目标

本图谱用于让开发者基于当前实现快速理解 Navia V1.2 的目标架构、当前实现状态、目标功能形态、开发阶段划分、验收门槛和最终用户体验路径。

图谱颜色约定：

| 颜色 | 含义 |
|---|---|
| 绿色 | 当前已实现，并已有自动化测试、截图、报告或代码证据 |
| 黄色 | 已有基础实现，但仍需要修改、质量深化或体验补强 |
| 蓝色 | 当前阶段尚未完成，需要新增阶段或后续实现 |
| 灰色 / 红色 | 声明边界、No-Go 或当前禁止越界范围 |

## 3. 设计口径

- A/C/D 是 service-layer 模块。
- B 是 frontend renderer 模块。
- D 是 CoreProvider 适配层和 Adapter 编排核心；piAgent 是默认首选 CoreProvider。
- Memory Plane 是 V2 预留位置，只能由 A 输出 `MemoryCandidate`，再由 D / Governance 决定是否沉淀为 `KnowledgeItem`；V1.2 不写长期记忆。
- Tool / Skill / MCP Plane 是 V4 预留位置，Web Research、PPT Agent、本地文件工具都必须经 D Adapter、Permission、Approval、Budget 和 Sandbox 调用；V1.2 禁止启用。
- Cloud / Multi-device Plane 是 V5 预留位置，不能破坏 Headless Runtime、Event、Trace 和 Artifact 合同。
- B 不直接调用 A/C/Core/MCP/Skill/API。
- A/C 不调用 piAgent。
- piAgent 不直接写 ArtifactRecord、SSE、EventStore 或 UI。
- Integration Codex 只负责 wiring、合同审计、真实 E2E 和 PRD 复检。
- 当前已实现基线是 `V1.2-AC-Native` closeout evidence passed，`V1.2-AC-Quality` evidence passed，且 `V1.2-AC-Jumpback MVP` evidence passed。
- 当前后续阶段进入 `V1.2-Closeout`：截图级 Jumpback、A SourceRef 质量、C 节点消歧、B Mindmap 交互细化、20 页真实网页矩阵和最终 PRD 复检。
- A high-signal 输出必须进入 Runtime 主链路后，C 才能声明 digest-first mindmap。
- C 节点反跳必须绑定 A SourceRef 或明确 paragraph/chunk fallback。
- 原生 Side Panel 是当前通过的体验容器；最终 V1 前端仍以网页内悬浮球和双轨面板为目标形态。

## 4. 验收口径

图谱必须能回答：

- 当前实现和目标架构差异是什么？
- 每个模块内部有哪些组件，尤其 D 的 CoreProvider Layer 与 Adapter Layer 如何分层？
- 模块间公共 API、内部合同和 CoreProvider API 如何调用？
- 关键用户体验路径的数据如何流转？
- 哪些测试和证据可以证明 V1.2 完成？
- 什么时候必须 No-Go？
- 后续 V2-V5 哪些架构位置可变，哪些职责边界必须保持稳定？
- 四位开发如何并行推进 A/B/C/D，哪些交付物必须通过公共合同进入 Integration？
- 当前阶段 A/C 还差什么才能从“模块可用”变成“产品链路可验收”？
- 读取网页 -> A JSON -> C Mermaid -> source fallback 的证据链是否完整？
- Mindmap 节点点击后是否能映射到 nodeSourceMap、展示来源证据卡片，并在 DOM 反跳失败时可读降级？
- 当前能声明完成什么，不能声明完成什么？
- V1.2-Closeout 是否具备足够截图、metadata、真实网页矩阵和 false-green audit 来支撑 V1.2 完成声明？
