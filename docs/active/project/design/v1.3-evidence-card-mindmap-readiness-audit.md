## 3. 外部审查发现

### 3.1 已关闭的问题

- V1.3 目标已从“体验基线”明确为本阶段正式开发目标。
- 文档已区分“当前基线存在”和“V1.3 出门验收完成”。
- drawio 已去除“20 页 / 5 个样本已完成”这类过强表述，改为 V1.3 真实出门条件。
- PRD、架构、开发计划、验收计划和 stage gate 已统一使用 `V1.3-0` 到 `V1.3-5`。
- V1.3 合同 schema 已补入现有 report 使用的 `generatedAt` 与 `visualEvidenceStatus` 字段，避免 schema 与报告结构脱节。
- V1.3-1 已明确必须使用 `$defs.EvidenceCardViewModel` 单独验证 fixture，不能只验证最终 acceptance report。
- V1.3-5 已明确需要 report semantic validation，不能只依赖 JSON Schema 字段校验。

### 3.2 仍需在 V1.3-0 关闭的事项

- 对现有 `evidence/v1_3_evidence_card_mindmap/report.json` 重新做 schema validation。
- 对现有 `evidence/v1_3_evidence_card_mindmap/report.json` 重新做 semantic validation。
- 检查 `report.json` 中 runtime snapshot page 与 native Side Panel screenshot sample 的关系，避免把 `visualEvidenceStatus="not_sampled"` 页面误读成视觉样本。
- 复核现有 PRD review 和 false-green audit 是否足够详细；当前版本偏简短，V1.3-5 出门前应扩充为逐项矩阵。
- 人工审核 drawio 是否存在方向偏移或过度计划承诺。
- 明确 Chrome 原生 Side Panel E2E 若因本机浏览器依赖阻塞时，哪些证据可作为 blocker，哪些不能计入 pass。

Semantic validation 最低规则：

- `passed=true` 时 `fatalIssues` 和 `majorIssues` 必须为空。
- `pagesPassed` 不得大于 `pagesTotal`，且必须满足 active gate。
- `nativeSidePanelSamples` 必须能追溯到 `screenshots[]` 中真实 native Side Panel 样本。
- `visualEvidenceStatus="not_sampled"` 的页面不得计入 native Side Panel visual sample。
- fallback evidence 不得计入 DOM highlight success。

## 4. 开发前 Go / No-Go

Go 条件：

- `V1.3-0` 文档一致性审计无 fatal / major。
- drawio 经人工审核，确认目标没有偏离当前阶段。
- report schema 与现有证据格式可解析、可验证。
- Stage gate 明确“V1.3 只做 Evidence Card Mindmap，不做最终网页内双轨面板”。

No-Go 条件：

- drawio 或 PRD 暗示完整 V1 complete。
- 把现有证据包的 `passed=true` 直接等同于更新后 V1.3 complete。
- 只用全屏 extension page 或组件测试替代 native Side Panel 截图样本。
- fallback evidence 被标记为 DOM highlight success。
- B 生成事实、Mindmap 或 Artifact；C 输出前端组件结构。
