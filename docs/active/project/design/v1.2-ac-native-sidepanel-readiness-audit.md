## 3. 关键开发目标

### 3.1 Native 入口稳定

目标：

- 在普通网页中通过 Chrome extension action 或快捷键稳定打开 Navia 原生 Side Panel。
- 不再使用全屏 `chrome-extension://.../sidepanel.html` 作为用户体验通过依据。

验收：

- 截图同时包含真实网页主体内容和右侧 Navia Side Panel。
- 如果 Chrome 自动化无法稳定定位原生 Side Panel，必须输出 structured blocker。

### 3.2 Side Panel 窄宽度可用

目标：

- `读取当前页面`、`提交上下文`、`总结`、`Mindmap`、Debug 入口在 Side Panel 宽度内可见或可稳定到达。
- Runtime offline、PageContext missing、tool failure、Mermaid render failure 在 Side Panel 内可见。

验收：

- 真实 Side Panel 截图展示关键入口。
- 失败态有可见文案，不得静默失败。

### 3.3 AC 功能链路在 Native 中可验收

目标：

- 同一 Side Panel 中完成读取当前页、Debug、页面总结、页面问答、Mindmap 和 source fallback。
- A/C/D/B 边界不变。

验收：

- A 不创建 Artifact、不发 SSE、不写 EventStore / Trace。
- C 不读取 DOM。
- D 仍是 ToolResult / Artifact / Event / Trace 唯一映射出口。
- B 不绕过 Runtime / D 直接生成总结、回答或 Mindmap。

## 4. False-Green 风险

| 风险 | 防线 |
|---|---|
| 全屏插件页冒充原生 Side Panel | 通过截图必须同时包含真实网页和右侧 Navia Side Panel |
| probe 打开成功被误当完整 UX 通过 | `native-probe` 只能证明打开状态，完整通过必须来自 `native-ux` 或人工等价截图 |
| 坐标点击导致误截无关窗口 | 自动化需要稳定 target / test id；否则输出 blocker |
| 截图证据无法追溯 | 每张通过截图必须有同名 metadata JSON，记录 URL、页面标题、extensionId、native 判定、viewport、panel 宽度、runtime 状态和结论 |
| blocker 只是自然语言描述，无法机器检查 | blocker 必须按 `NativeSidePanelBlocker` 最小字段输出，并在 `blocksCompletion=true` 时阻断完成声明 |
| probe / UX 报告混在一起 | `native-sidepanel-probe/report.json` 与 `native-sidepanel-ux/report.json` 必须分层，只有 UX 报告或人工等价截图可触发体验通过 |
| Debug 只显示大 JSON，无法人工判断 | Debug 必须解释 activePage、A quality、source map 或失败原因 |
| low-signal 页面被伪装成正常结果 | low-signal 页面必须 degraded / fail 且 Side Panel 可见 |
| 前端绕过 Runtime 造结果 | B 不得直接生成 summary、answer 或 mindmap |

## 5. 开发前 Go / No-Go

Go 条件：

- `V1.2-AC-Native-0` 外部审计无 fatal / major。
- 当前文档包已被开发者阅读并作为唯一阶段依据。
- 真实 Chrome / Chromium 验收路径、截图标准、证据目录和 No-Go 已冻结。

No-Go 条件：

- 外部审计指出 fatal / major 规格偏差未闭环。
- direct extension page 仍被计划作为体验通过依据。
- 自动化无法区分 native panel 与全屏插件页。
- 开发计划试图在本阶段引入 RAG、长期记忆、多 Agent、浏览器自动操作或其他 V1 禁止范围能力。
