## 1. 图谱目标

这份图谱只服务 `V1.2-AC-Native` 阶段，不替代 V1.2 总体架构图。

目标是让开发者和审计者快速看清：

- 当前 AC 功能桥接与真实 Chrome 原生 Side Panel 用户体验之间的差距。
- 目标架构中 Web Page、Chrome Native Side Panel、B Renderer、Runtime、D Adapter、A/C 服务之间的调用关系。
- 用户关键体验路径如何从真实网页进入 Side Panel，再回到 Runtime 和 A/C/D。
- 每个子阶段的开发计划、验收门槛、出门条件和 No-Go。

---

## 2. 页面结构

| 页 | 名称 | 用途 |
|---|---|---|
| 1 | 当前差异与阶段目标 | 对比 direct page smoke、native probe blocker 和目标用户体验 |
| 2 | 目标架构与调用路径 | 画出真实网页 -> 原生 Side Panel -> Runtime -> D -> A/C -> B 的完整路径 |
| 3 | 开发与验收计划 | 展示 `V1.2-AC-Native-0` 到 `V1.2-AC-Native-5` 的阶段门禁 |
| 4 | 用户体验路径与截图标准 | 明确用户操作路径和截图证据必须包含的元素 |
| 5 | 出门条件与 No-Go | 固化 native-only 通过条件和 false-green 防线 |

---

## 3. 使用规则

- 本图只作为 V1.2-AC-Native 开发与审计依据。
- direct extension page 只能作为功能 smoke，不得在本图中被标记为用户体验通过路径。
- 若后续要变更 A/C/D 公共合同，必须回到 V1.2-0 或对应模块合同冻结，不得通过本图直接扩大范围。
- 如果 Chrome 自动化无法稳定操作原生 Side Panel，必须输出 blocker 和人工验收路径，不得把不确定截图标记为 pass。
- 本阶段截图证据必须配套 metadata JSON；自动化失败必须配套 structured blocker JSON。
- `native-probe` 与 `native-ux` 报告必须分层，probe 通过不能单独触发体验通过。
