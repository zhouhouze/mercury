# V1.2 AI 伴读工作区划分与独立开发协议

版本：V1.2 Workspace Partition Draft
日期：2026-06-04
适用范围：V1.2 扩展文档阶段与后续 A/B/C/D 独立 Codex 终端开发

---

## 1. 目标

V1.2 扩展阶段的目标不是立即写代码，而是先冻结 AI 伴读四模块的职责、合同和工作区边界，使后续四个 Codex 终端可以独立开发并只在各自工作区内修改代码和完成验收。

四个开发模块为：

```text
A：网页信息提取、过滤、蒸馏与结构化总结
B：结构化数据、流式文本、Mindmap 的前端实时渲染
C：基于网页结构化 JSON 的思维导图生成
D：CoreProvider + Adapter Layer，负责可替换 Core 适配、治理桥与 Adapter 编排
```

本协议解决三个问题：

- 每个模块改哪里。
- 每个模块输出什么。
- 什么时候必须停止开发并回到合同文档阶段。

V1.2 工作区重组原则：

- A/C/D 是 service-layer 能力，集中在 `services/local-runtime/navia_runtime/modules/`。
- B 是 frontend-rendering 能力，集中在 `apps/chrome-extension/src/modules/`。
- 每个模块在所属层内有独立文件夹。
- 模块自己的功能代码、模块文档、合同说明和测试说明都放在对应文件夹内。
- 既有入口文件只作为 Integration wiring 层，不再作为 A/B/C/D 模块长期工作区。
- 后续代码迁移必须先把实现落到模块目录，再由 Integration Codex 接入既有入口。

---

## 2. 阶段边界

V1.2 扩展允许定义轻量 Adapter 合同：

- MCP Adapter。
- Skill Adapter。
- External API Adapter。
- Internal Tool Adapter。

但 V1.2 扩展仍禁止：

- 长期记忆。
- 完整 RAG。
- 多 Agent 编排。
- 浏览器自动点击、自动导航或自动操作。
- 默认读取本地文件。
- 高风险 MCP / Skill side effect 自动执行。
- 联网搜索作为默认伴读能力。
- 深度研究、PPT 生成、语音、桌宠。

V1.2 的 D 模块只支持单 Session 连续上下文与 checkpoint，不实现跨网页长期知识库。D 模块采用 `CoreProvider` 可替换策略，`piAgentProvider` 是首选实现目标，`MockCoreProvider` 是合同测试和 fallback；真实 piAgent 接入前必须锁定仓库、版本或 commit、license、运行时和工具调用模型。

## 3. 总体分工

### 3.1 Canonical module folders

V1.2 后续实质开发统一使用以下目录：

```text
services/local-runtime/navia_runtime/modules/page_reading/
services/local-runtime/navia_runtime/modules/mindmap/
services/local-runtime/navia_runtime/modules/agent_loop/
services/local-runtime/navia_runtime/modules/adapters/
apps/chrome-extension/src/modules/chat_renderer/
apps/chrome-extension/src/modules/artifact_renderer/
apps/chrome-extension/src/modules/debug_renderer/
apps/chrome-extension/src/modules/mindmap_renderer/
```

Service 模块目录内部采用统一结构：

```text
<module>/
  README.md
  docs/
  contracts/
  runtime/
  tests/
  fixtures/
```

Frontend 模块目录内部采用统一结构：

```text
<module>/
  README.md
  docs/
  tests/
  fixtures/
```

目录语义：

- `runtime/`：Python Runtime 侧实现，仅 service 模块使用。
- `docs/`：模块实现说明、审计记录和验收证据。
- `contracts/`：模块内部合同，不能替代公共合同。
- `tests/`：该模块自己的单测、合同测试和验收脚本。
- `fixtures/`：该模块独立使用的真实数据 fixture。

### 3.2 Flow

```text
Chrome Page DOM
  |
  v
A Page Reading Workspace
  -> StructuredPageContext
  -> ParagraphAnnotation[]
  -> PageChunk[]
  |
  v
D CoreProvider / Adapter Workspace
  -> AdapterRegistry
  -> ToolResult
  -> AgentEvent SSE
  -> ArtifactRecord
  |
  +--> C Mindmap Workspace
  |      -> Mermaid source
  |      -> source paragraph/chunk links
  |
  v
B Chat Renderer Workspace
  -> streaming response UI
  -> structured summary UI
  -> mindmap UI
  -> source fallback / debug UI
```

关键原则：

- A 产生结构化网页事实。
- C 消费 A 的结构化事实生成 Mindmap。
- D 统一编排 A/C/API/MCP/Skill Adapter。
- B 只消费 D 的事件流和 artifacts。
- 前端不得绕过 D 直接调用 A/C/MCP/Skill/API。

## 4. A 模块：网页信息提取、过滤、蒸馏与结构化总结

### 4.1 职责

A 模块负责将网页转成高密度结构化 JSON：

- DOM 文本提取。
- `cleanedText` 生成。
- heading tree。
- paragraph blocks。
- semantic chunks。
- 段落标注。
- 信息密度评分。
- 重要性评分。
- 结构化 summary draft。

### 4.2 允许工作区

后续实质开发时，A 模块 Codex 只允许修改：

```text
services/local-runtime/navia_runtime/modules/page_reading/
docs/navia_v1_project_docs/stage-gates/v1.2-a-page-reading.md
```

说明：

- A 的功能代码必须全部落在 `services/local-runtime/navia_runtime/modules/page_reading/`。
- `apps/chrome-extension/src/pageContext.ts` 是既有入口，不归 A 模块直接修改；需要接入时由 Integration Codex wiring。
- 如果需要修改 `contracts.py`、`06-api-contract.md` 或 `07-data-models.md`，必须停止开发，回到 V1.2-0 合同冻结阶段。

### 4.3 输出合同

A 输出：

```text
StructuredPageContext
ParagraphAnnotation[]
PageChunk[]
StructuredSummaryDraft optional
```

必须包含：

```text
pageId
url
title
domain
contentHash
capturedAt
headingTree
paragraphs
chunks
annotations
```

## 5. B 模块：前端实时渲染

### 5.1 职责

B 模块负责 Chat 页签内所有实时渲染：

- SSE `response.delta` 流式文本。
- structured summary 展示。
- paragraph annotation 展示。
- tool running / tool failure 展示。
- artifact 卡片。
- Mermaid visual render。
- Mermaid source fallback。
- Debug 诊断面板。

### 5.2 允许工作区

后续实质开发时，B 模块 Codex 只允许修改：

```text
apps/chrome-extension/src/modules/chat_renderer/
apps/chrome-extension/src/modules/artifact_renderer/
apps/chrome-extension/src/modules/debug_renderer/
apps/chrome-extension/src/modules/mindmap_renderer/
docs/navia_v1_project_docs/stage-gates/v1.2-b-chat-renderer.md
```

说明：

- B 的功能代码必须全部落在 `apps/chrome-extension/src/modules/*_renderer/`。
- `apps/chrome-extension/src/injectedPanel.ts`、`sse.ts` 和 E2E 入口不归 B 模块直接修改；需要接入时由 Integration Codex wiring。
- B 不得修改 Runtime AgentCore、ToolExecutor、AdapterRegistry 或 PageContext 合同。

### 5.3 输入合同

B 只消费：

```text
AgentEvent SSE
ArtifactRecord
RuntimeStatus
Session restore response
```

B 不得直接调用：

```text
A page extraction runtime service
C mindmap generation runtime service
MCP server
Skill runner
External API
```

## 6. C 模块：思维导图生成

### 6.1 职责

C 模块负责从 A 的结构化网页 JSON 生成可追溯 Mindmap：

- 从 heading tree 和 high-importance chunks 生成 Mermaid。
- Mermaid validation。
- 最多一次 repair。
- Mindmap 节点绑定 source paragraph/chunk。
- 节点回跳 metadata。

### 6.2 允许工作区

后续实质开发时，C 模块 Codex 只允许修改：

```text
services/local-runtime/navia_runtime/modules/mindmap/
docs/navia_v1_project_docs/stage-gates/v1.2-c-mindmap.md
```

说明：

- C 的功能代码必须全部落在 `services/local-runtime/navia_runtime/modules/mindmap/`。
- `services/local-runtime/navia_runtime/tools.py` 中的既有 `generate_mindmap` 入口不归 C 模块直接修改；需要接入时由 Integration Codex wiring。

### 6.3 输出合同

C 输出：

```text
ArtifactRecord(type="mindmap", metadata.format="mermaid")
```

必须包含：

```text
content = Mermaid source
sourcePageId
sourceChunkIds
metadata.paragraphIds
metadata.validation
metadata.nodeSourceMap
```

“反跳功能”定义：

- 用户点击 Mindmap 节点时，B 模块可以根据 `nodeSourceMap` 找到源 paragraph/chunk。
- 若当前网页仍可访问，前端可滚动并高亮源段落。
- 若无法定位 DOM，前端必须展示 source excerpt fallback。

## 7. D 模块：CoreProvider + Adapter Layer

### 7.1 职责

D 模块是 Navia Runtime 与可替换 Agent Core Provider 之间的稳定适配层。复杂 Core 能力由 `piAgentProvider`、`MockCoreProvider` 或后续 `FutureCoreProvider` 承接；D 负责构造 Core 输入、执行治理、调用 Adapter、映射输出和维护 Trace。

职责：

- 创建 turn、trace、request 相关 ID。
- 管理单 Session 连续上下文、recent messages 和 checkpoint。
- 读取 activePage、结构化网页结果和历史消息摘要。
- 构造 `CoreTurnInput` 并调用 `CoreProvider.run_turn()`。
- 使用 `MockCoreProvider` 做合同测试和自动化 fallback。
- 为 `piAgentProvider` 提供适配壳；真实接入前必须锁定 piAgent 仓库、版本或 commit、license、运行时和工具调用模型。
- 执行 StateMachine。
- 执行 Budget / Permission / Context supervision。
- 通过 GovernanceBridge 确保 Core 请求工具前完成预算和权限检查。
- 调用 AdapterRegistry。
- 统一映射 ToolResult、ArtifactRecord、AgentEvent SSE 和 Trace。

### 7.2 允许工作区

后续实质开发时，D 模块 Codex 只允许修改：

```text
services/local-runtime/navia_runtime/modules/agent_loop/
services/local-runtime/navia_runtime/modules/adapters/
docs/navia_v1_project_docs/stage-gates/v1.2-d-agentic-loop.md
```

说明：

- D 的功能代码必须全部落在 `services/local-runtime/navia_runtime/modules/agent_loop/` 和 `services/local-runtime/navia_runtime/modules/adapters/`。
- `services/local-runtime/navia_runtime/agent.py` 和 `tools.py` 是既有入口，不归 D 模块直接修改；需要接入时由 Integration Codex wiring。
- D 不得修改 B 的 UI 实现。
- D 不得直接变更 A/C 输出合同；需要变更时回到 V1.2-0。

### 7.3 接入规则

A/C/MCP/Skill/API 必须通过 D 注册为 Adapter：

```text
AdapterSpec
-> AdapterRegistry
-> GovernanceHooks
-> AdapterInvocation
-> ToolResult
-> ArtifactRecord optional
-> AgentEvent SSE
```

D 不得允许：

- 前端直接调用 MCP。
- 前端直接调用 Skill。
- Adapter 绕过 governance hooks。
- Adapter 返回自由文本而不是 ToolResult。
- 高风险 side effect 默认执行。
- CoreProvider 直接写 ArtifactRecord、SSE、EventStore、Trace 或 UI。

## 8. Integration Codex

除 A/B/C/D 四个开发终端外，必须设置 Integration Codex。

Integration Codex 职责：

- 不开发模块业务逻辑。
- 只负责 wiring。
- 只负责合同一致性审计。
- 只负责端到端验收。
- 只负责 trace / artifact / SSE / PRD 复检。

允许工作区：

```text
services/local-runtime/navia_runtime/app.py
services/local-runtime/navia_runtime/contracts.py
services/local-runtime/navia_runtime/stores.py
services/local-runtime/navia_runtime/agent.py
services/local-runtime/navia_runtime/tools.py
apps/chrome-extension/src/pageContext.ts
apps/chrome-extension/src/injectedPanel.ts
apps/chrome-extension/src/sse.ts
apps/chrome-extension/src/runtimeClient.ts
services/local-runtime/tests/integration/
apps/chrome-extension/e2e/
docs/navia_v1_project_docs/stage-gates/v1.2-e-integration.md
```

如果 Integration Codex 发现合同冲突，不得自行重写模块，应打回对应模块或回到 V1.2-0。

## 9. 跨模块变更流程

任一模块如果需要修改以下内容，必须停止开发：

```text
06-api-contract.md
07-data-models.md
contracts/*.schema.json
public Runtime endpoint
AgentEvent type
ToolResult envelope
ArtifactRecord shape
StructuredPageContext shape
AdapterSpec shape
canonical module folder
```

处理流程：

```text
发现合同变更需求
-> 停止模块开发
-> 更新 V1.2-0 stage-gate
-> 更新合同文档
-> 做合同审计
-> 无致命/重大风险后恢复模块开发
```

## 11. V1.2-0 Go / No-Go

Go 条件：

- A/B/C/D 工作区边界明确。
- Integration Codex 职责明确。
- Adapter 合同明确。
- 跨模块合同变更流程明确。
- 每个模块都有输入、输出和禁止事项。
- 后续 Codex 终端可以独立开发，不需要自行决定目录边界。

No-Go 条件：

- 任一模块需要频繁修改其他模块目录。
- B 可以绕过 D 直接调用工具或外部服务。
- MCP / Skill / API 可以绕过 D。
- C 的 Mindmap 节点不能追溯到 A 的 paragraph/chunk。
- D 的历史对话范围变成长期记忆或 RAG。
- 没有 Integration Codex 负责最终 wiring 和验收。
