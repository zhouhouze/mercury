版本：V1.2 Module-local Design Package
日期：2026-06-04
状态：Documentation implementation complete

## 1. 目标

本文件登记 V1.2 各工作区内的深度设计文档包。后续 A/B/C/D 模块开发者应优先阅读自己工作区内的文档，再参考公共 PRD、API 合同和数据模型。

目标是让每个模块可以独立实现和 mock 验证，最后由 Integration Codex 做少量 wiring 联调。

## 3. 工作区入口

### A Page Reading

```text
services/local-runtime/navia_runtime/modules/page_reading/
```

开发者只读本模块文档即可理解：

- `StructuredPageContext` 生成流程。
- heading / paragraph / chunk / annotation 规则。
- deterministic mock 标注策略。
- 真实 HTML fixture 验收。
- 与 D/C/B 的边界。

### B Frontend Renderer

```text
apps/chrome-extension/src/modules/chat_renderer/
apps/chrome-extension/src/modules/artifact_renderer/
apps/chrome-extension/src/modules/debug_renderer/
apps/chrome-extension/src/modules/mindmap_renderer/
```

开发者只读本模块文档即可理解：

- SSE 到 UI presentation state 的映射。
- artifact card 渲染。
- Mermaid visual / source fallback。
- Debug tab 诊断边界。
- B 不拥有 AgentCore state 的约束。

### C Mindmap

```text
services/local-runtime/navia_runtime/modules/mindmap/
```

开发者只读本模块文档即可理解：

- `StructuredPageContext` 到 Mermaid source 的生成流程。
- `nodeSourceMap` 和 source fallback。
- Mermaid validation / at-most-once repair。
- C 不负责 UI 渲染和 SSE。

### D AgenticLoop / Adapters

```text
services/local-runtime/navia_runtime/modules/agent_loop/
services/local-runtime/navia_runtime/modules/adapters/
```

开发者只读本模块文档即可理解：

- chat turn 编排。
- CoreProvider 可替换策略。
- piAgentProvider 首选实现目标。
- MockCoreProvider 测试 fallback。
- AdapterRegistry。
- governance hooks。
- ToolResult / ArtifactRecord / AgentEvent 映射。
- MCP / Skill / API Adapter 风险边界。

## 4. Integration Codex 职责

Integration Codex 是唯一跨模块 wiring 角色。它负责：

- 将 A 接入现有 PageContext / Runtime session。
- 将 C 接入 D 的 internal adapter。
- 将 D 接入 `/v1/chat/stream` 和 trace。
- 将 B 接入 existing injected panel / SSE client。
- 执行跨模块 E2E 和 PRD 覆盖验收。

Integration Codex 不应重写 A/B/C/D 模块业务逻辑。

## 5. Go / No-Go

Go 条件：

- 每个模块文档包存在。
- 每个模块都有 mock 验收计划。
- 每个模块都声明 PRD 覆盖与非覆盖项。
- 每个模块都声明 integration boundary 和 stop conditions。

No-Go 条件：

- 模块文档要求跨目录直接修改。
- B 直接调用 A/C/MCP/Skill/API。
- A/C 绕过 D 直接发 SSE 或创建 UI 状态。
- D 绕过 governance 执行 adapter。
- Adapter 默认执行高风险 side effect。

## 6. 审计结论

V1.2 模块内深度设计文档包已满足“独立开发 + 实际 mock 验证 + 少量联调”的文档目标。进入实质开发前仍需按各 stage-gate 逐阶段完成 PRD 规格检视和模块 mock 验收。

补充约束：

- `public-api.md` 是模块实现入口的默认合同。
- `executable-contract.md` 是模块测试必须验证的最低断言。
- `fixture-spec.md` 是 mock 验收和 evidence 的输入来源。
- `test-and-evidence-plan.md` 是模块退出验收的证据要求。
