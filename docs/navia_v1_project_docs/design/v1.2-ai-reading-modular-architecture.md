# V1.2 AI 伴读四模块目标架构

版本：V1.2 Architecture Draft
日期：2026-06-03
适用范围：V1.1 之后的 AI 伴读能力深化与 ChatBox 架构拆分

---

## 1. 背景与目标

当前“聊天”能力已经完成 V1 主链路雏形：

```text
Chrome 页面内面板
-> PageContext 抽取
-> Local Runtime
-> /v1/chat/stream
-> TurnRunner
-> IntentRouter
-> ToolExecutor
-> SSE response / Artifact / Trace
```

但现有能力仍表现为一个大 ChatBox：网页读取、结构化总结、流式渲染、Mindmap 生成和 AgenticLoop 编排的边界不够清晰。V1.2 的设计目标是把 AI 伴读拆成四个模块，让后续开发可以独立增强每个模块，而不破坏 V1 已冻结的 API / Event / Tool / Artifact 合同。

四个模块为：

1. 网页数据抓取与结构化总结。
2. 基于网页内容的流式渲染。
3. 基于网页内容的思维导图生成。
4. AgenticLoop 实现 ChatBox。

本设计不扩大 V1 范围，不引入 RAG、长期记忆、多 Agent、浏览器自动操作、联网搜索、语音、桌宠、深度研究或 PPT 生成。V1.2 扩展允许定义轻量 MCP / Skill / API Adapter 合同，但所有 Adapter 必须通过 D 模块 AgenticLoop 和 governance hooks 接入，不允许直连或默认执行高风险 side effect。

---

## 2. 总体架构

```text
Chrome Tab DOM
  |
  | content script / pageContext.ts
  v
Page Reading & Structured Summary Module
  |  POST /v1/page/context
  v
Local Runtime Session.activePage
  |
  | POST /v1/chat/stream
  v
AgenticLoop ChatBox Module
  |-- IntentRouter
  |-- StateMachine
  |-- Governance hooks
  |-- AdapterRegistry
  |-- ToolExecutor
  |       |-- summarize_page
  |       |-- answer_from_page
  |       |-- explain_selection
  |       |-- generate_mindmap
  |       |-- mcp adapters
  |       |-- skill adapters
  |       `-- external api adapters
  |
  | SSE AgentEvent
  v
Page-grounded Streaming Renderer Module
  |
  | artifact.created
  v
Mindmap Generation / Artifact Renderer
```

架构原则：

- Runtime 是 AgentCore 状态唯一事实源。
- Frontend 只维护展示态，不保存 AgentCore 核心状态。
- 所有工具必须消费真实 `session.activePage`。
- 所有工具必须返回 `ToolResult` envelope。
- 所有成功工具必须创建 `ArtifactRecord`。
- 每个 artifact 必须关联 `sourcePageId`、`turnId`、`toolCallId`。
- 所有执行中事件必须进入 EventStore，并可通过 trace API 审计。
- MCP / Skill / API Adapter 不得绕过 AgenticLoop、ToolExecutor 或 governance hooks。

## 3. 模块一：网页数据抓取与结构化总结

### 3.1 职责

该模块负责把当前网页变成 Runtime 可消费的结构化上下文，并生成页面级总结 artifact。

输入：

- 当前 Chrome tab 的 URL、title、domain。
- DOM 提取后的 `headings`、`selected_text`、`visible_text`、`cleaned_text`。
- tab id、捕获时间、可选 metadata。

输出：

- `PageContext`。
- `PageChunk[]`。
- `PageContextRef` 写入 `session.activePage`。
- `ArtifactRecord(type="summary", source="page", metadata.format="markdown")`。

### 3.2 边界

该模块不负责：

- ChatBox 的 turn 编排。
- SSE UI 渲染。
- Mindmap 视觉渲染。
- 长期知识库、RAG 或 Memory Plane。

### 3.3 结构化规则

`PageContext` 必须至少包含：

```text
pageId
sessionId
url
title
domain
capturedAt
contentHash
headings
selectedText optional
visibleText optional
cleanedText
chunks
```

chunk 规则：

- 每个 chunk 必须有 `chunkId`、`pageId`、`order`、`text`。
- 有标题路径时必须保留 `headingPath`。
- 长网页不得只保留一整段 `cleanedText`。
- `contentHash` 用于识别同一页面内容是否变化。

### 3.4 总结工具规则

`summarize_page` 必须：

- 从 `session.activePage` 读取真实网页内容。
- 缺少 activePage 时返回 `PAGE_CONTEXT_REQUIRED`。
- 不得生成假 summary 或假 artifact。
- 返回 `ToolResult(status="succeeded")`。
- 创建 `ArtifactRecord(type="summary")`。
- 通过 `artifact.created` 事件写入 EventStore。

## 4. 模块二：基于网页内容的流式渲染

### 4.1 职责

该模块负责前端消费 Runtime SSE，并把 AgentCore 执行过程转化为用户可见的 ChatBox 展示。

输入：

- `/v1/chat/stream` 返回的 SSE AgentEvent。
- Runtime health / models status。
- 当前 PageContext 提交状态。

输出：

- 用户消息。
- assistant 流式回答。
- 工具运行状态。
- artifact 卡片。
- offline、missing context、tool failure、Mermaid render failure 等可见状态。

### 4.2 必须支持的事件

```text
state.transition
page.context.received
intent.detected
budget.checked
budget.exhausted
tool.requested
tool.denied
tool.started
tool.done
tool.failed
response.delta
response.done
artifact.created
error
```

未知事件处理：

- 不得导致 UI 崩溃。
- 可写入 debug 面板。
- 不得改变 Runtime AgentCore 状态。

### 4.3 前端状态边界

前端可以维护：

- 面板展开/收起。
- 当前工具栏 tab。
- 输入框内容。
- 消息滚动位置。
- 当前 Runtime online/offline 展示态。
- artifact 展示态。

前端不得维护：

- Agent turn 状态机。
- ToolCallRecord 最终状态。
- BudgetLedger。
- ArtifactRecord 的事实状态。
- Session trace。

## 5. 模块三：基于网页内容的思维导图生成

### 5.1 职责

该模块把当前网页结构转为 Mermaid mindmap artifact，并在前端展示。

Runtime 负责：

- 从 `session.activePage` 读取 headings、chunks、summary。
- 生成 Mermaid source。
- Mermaid source 校验。
- 最多一次 repair。
- 创建 `ArtifactRecord(type="mindmap", metadata.format="mermaid")`。

Frontend 负责：

- 渲染 Mermaid。
- 渲染失败时展示 source fallback。
- 将 Mermaid render failure 作为 UI 可见错误处理，不影响 ChatBox 继续对话。

### 5.2 生成策略

V1.2 默认采用 deterministic fallback：

```text
headings -> bounded tree -> Mermaid mindmap
```

约束：

- 最大深度：4。
- 最大节点数：40。
- repair 次数：最多 1 次。
- Runtime 不声明 Mermaid 视觉验收通过，视觉验收由前端 renderer 和 E2E 截图负责。

### 5.3 事件与数据承载

不得新增 ad-hoc Mermaid 事件类型。

Mermaid validation / repair 详情写入：

- `tool.done.data.validation`。
- `tool.done.data.repair`。
- `artifact.created.data.metadata`。
- `ArtifactRecord.metadata.validation`。

失败时使用：

```text
event type = error
error.code = MERMAID_VALIDATION_FAILED
```

## 6. 模块四：AgenticLoop ChatBox

### 6.1 职责

AgenticLoop ChatBox 是 Runtime 中的 turn 编排模块。它不等于前端聊天 UI，而是每次用户输入后的执行闭环。

职责：

- 创建 `turn_id`、`trace_id`、`request_id`。
- 写入 user message。
- 通过 StateMachine 执行状态迁移。
- 调用 IntentRouter。
- 执行 ContextSupervisor / BudgetSupervisor / ToolPermissionSupervisor。
- 通过 ToolExecutor 调用工具。
- 通过 AdapterRegistry 接入内部工具、MCP、Skill 和 External API Adapter。
- 持久化 ToolCallRecord、ArtifactRecord、BudgetLedger、AgentEvent。
- 通过 SSE 输出 AgentEvent。

### 6.2 主路径

```text
waiting_user
-> detecting_intent
-> planning
-> budget_checking
-> running_tool
-> streaming_response
-> persisting_turn
-> waiting_user
```

每个状态迁移必须：

- 经过 `StateMachine.transition()`。
- 产生 `state.transition` event。
- 写入 EventStore。

### 6.3 工具执行约束

ToolExecutor 必须：

- 在 `tool.started` 前完成 budget check。
- 在 `tool.started` 前完成 permission check。
- 执行 PreToolUse hook。
- 执行 PostToolUse hook。
- 返回 ToolResult envelope。
- 禁止工具绕过 Runtime 直接向 UI 写结果。
- 禁止 MCP / Skill / API Adapter 绕过 D 模块直接被前端调用。

缺少 activePage 时：

- 产生 `error` event。
- 使用 `PAGE_CONTEXT_REQUIRED`。
- 不产生 `tool.started`。
- 不产生假 artifact。

## 7. 代码演进建议

现有代码可以先不破坏性重写，后续按以下顺序拆分：

```text
services/local-runtime/navia_runtime/tools.py
  -> reading_tools.py
  -> mindmap_tools.py
  -> tool_executor.py
  -> adapters.py

services/local-runtime/navia_runtime/agent.py
  -> agent.py 保留 public TurnRunner
  -> turn_loop.py 承载 AgenticLoop 编排
  -> response_stream.py 承载 SSE response 组织

apps/chrome-extension/src/injectedPanel.ts
  -> chatRenderer.ts
  -> debugPanel.ts
  -> artifactRenderer.ts
```

拆分原则：

- 外部 API 不变。
- 测试先行，先证明行为不变。
- 不在同一 PR 同时改 Runtime 合同和前端视觉。
- 每次拆分后必须跑 Runtime tests、extension tests、真实 Chrome E2E。
- A/B/C/D 四个模块的实质开发必须遵守 `v1.2-ai-reading-workspace-partition.md` 的工作区边界。

## 8. 验收标准

### 8.1 端到端主链路

```text
打开真实网页
-> 点击读取网页
-> Runtime 写入 session.activePage
-> 用户发送“总结当前网页”
-> SSE 输出 state / intent / budget / tool / artifact / response events
-> 前端流式显示总结
-> 用户发送“生成思维导图”
-> Runtime 生成 Mermaid artifact
-> 前端渲染 Mermaid 或展示 source fallback
-> trace?turn_id 可还原完整执行路径
```

### 8.2 模块验收

网页数据抓取与结构化总结：

- [ ] `PageContext` 含真实 URL、title、domain、contentHash、headings、chunks。
- [ ] `summarize_page` 使用 `session.activePage`。
- [ ] 缺少 activePage 返回 `PAGE_CONTEXT_REQUIRED`。
- [ ] 成功总结创建 summary artifact。

流式渲染：

- [ ] 前端消费 `/v1/chat/stream` SSE。
- [ ] `response.delta` 逐步追加。
- [ ] 未知 SSE event 不崩溃。
- [ ] offline、missing context、tool failure 可见。

思维导图：

- [ ] Runtime 生成 Mermaid source。
- [ ] Mermaid validation 记录到 tool/artifact metadata。
- [ ] 成功创建 mindmap artifact。
- [ ] 前端渲染失败时展示 source fallback。

AgenticLoop：

- [ ] 每个 turn 有 `turn_id`。
- [ ] 每个 tool call 关联 `session_id`、`turn_id`。
- [ ] 每个 artifact 关联 `sourcePageId`、`turnId`、`toolCallId`。
- [ ] 每次 state transition 写入 EventStore。
- [ ] trace 可按 `turn_id` 过滤。

### 8.3 No-Go 条件

任一出现即不得声明 AI 伴读架构完成：

- 前端直接生成摘要、回答或 mindmap。
- 工具未消费真实 `session.activePage`。
- 缺少 activePage 仍生成结果。
- 工具直接返回自由文本。
- 成功工具没有 artifact。
- artifact 缺少 `turnId` 或 `toolCallId`。
- 只有实时 SSE，没有 EventStore trace。
- 新增 ad-hoc event string 但未更新合同。
- MCP / Skill / API 绕过 D 模块 AgenticLoop 或 governance hooks。
- 借本次架构引入 RAG、长期记忆、多 Agent、浏览器自动操作或高风险 side effect 默认执行。

## 9. 与 V2 的升级关系

V1.2 不实现 V2 知识库，但必须给 V2 留出升级路径：

```text
PageContext
-> PageChunk
-> ArtifactRecord
-> Session Trace
-> MemoryCandidate
-> KnowledgeItem
```

V1.2 只要求 PageContext、Chunk、Artifact、Trace 足够结构化。`MemoryCandidate` 和 `KnowledgeItem` 不在 V1.2 实现范围。
